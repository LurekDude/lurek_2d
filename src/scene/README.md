# `src/scene/` — Scene Stack and Transitions

## Purpose

The scene module implements a push-down automaton for game state management —
the industry-standard pattern for navigating between a game's distinct modes
(title screen, main gameplay, pause menu, inventory screen, game-over).  Scenes
are pushed onto a stack; the top scene is active; popping a scene returns to
the one below it unchanged.  Animated transitions (fade-to-black, slide left,
slide right, slide up, slide down) bridge between states so state changes look
intentional rather than abrupt.

The push-down semantics make sub-states trivial: pushing a pause menu keeps
gameplay alive and paused underneath it; popping the pause menu resumes
exactly where the player was with all entity positions, timers, and scripted
state intact.  Per-scene data storage lets each scene maintain its own private
namespace of Lua values without polluting global variables — score, camera
offset, and wave counter are keyed by `SceneId` and accessible only while
that scene is on the stack.  Named scene templates (registered at startup)
allow scenes to be pushed by name rather than by pre-constructed table.

## Architecture

```
SceneStack (scene management)
  │
  ├── Stack operations
  │     ├── push(scene, transition, duration)
  │     ├── pop(transition, duration)
  │     └── switch_to(scene, transition, duration)
  │
  ├── Registry ── named scene templates
  │     ├── register_scene(name, callbacks)
  │     └── get_registered(name)
  │
  ├── Transitions
  │     ├── TransitionType: None, Fade, SlideLeft/Right/Up/Down
  │     └── ActiveTransition: progress tracking + completion
  │
  ├── Per-scene data storage
  │     └── set/get/has/remove_data(scene_id, key, value)
  │
  └── DepthSorter ── z-ordering for draw calls
        ├── add(depth, callback_index)
        ├── sort() → stable z-order
        └── flush() → drain in order
```

### How It Works

Scene IDs are monotonically increasing `u64` values generated by the
`SceneStack` and never reused.  Even if the same named template is pushed
twice, each push receives a distinct ID so per-scene data from a previous
run of a scene does not bleed into a fresh run.

Transitions are tracked via an `ActiveTransition` struct holding elapsed time
and total duration.  `is_transitioning()` returns `true` while the transition
is active; `transition_progress()` returns a 0.0–1.0 fraction that Lua draw
code uses to interpolate between the outgoing and incoming scene visuals.  The
engine does not perform built-in pixel compositing for transitions — Lua is
responsible for using the progress value to alpha-blend or slide draw calls,
giving games full artistic control over how transitions look.

The registry maps string names to `SceneId` values.  `push("gameplay")` looks
up the registered ID, then delegates to the standard `push(id, ...)` path.
Unregistered scene names produce a `LuaError::RuntimeError` at the push call
site rather than silently pushing nothing.

## Cross-Cutting Concerns

### Lua Integration

The Lua bridge lives in `src/lua_api/scene_api.rs` (~320 lines), exposing
scene management under `luna.scene.*`.

### Usage from Lua

```lua
-- Register scenes
luna.scene.register("title", { enter = title_enter, update = title_update, draw = title_draw })
luna.scene.register("gameplay", { enter = game_enter, update = game_update, draw = game_draw })

-- Navigate
luna.scene.push("title")
luna.scene.switchTo("gameplay", "fade", 0.5)

-- Per-scene data
luna.scene.setData("gameplay", "score", 0)
local score = luna.scene.getData("gameplay", "score")
```
