## Section 1 — Philosophy

- **Audience**: both (humans and AI).
- **Word budget**: ~150 words.
- **Content**:
  - AI-first: the CAG layer is designed for autonomous agents to navigate without human intervention; humans are the second-class reader.
  - No GUI editor: the Lurek2D engine has no embedded editor (binding constraint A-01); by symmetry, the CAG layer has no GUI tooling — all editing is GitOps via plain Markdown files.
  - GitOps: every CAG change is a commit; `tools/validate/cag_validate.py` is the merge gate; phases are recorded in `docs/CHANGELOG.md`.
  - Six personas (named — link forward to §4).
- **Linked templates**: `SYSTEM_PROMPT_TEMPLATE.md`.

## Section 6 — Authoring Guides

- **Audience**: humans primarily (new contributors).
- **Word budget**: ~300 words split into three sub-guides.
- **Content**:
  - **Add a new agent** — checklist: copy `AGENT_TEMPLATE.md` reference implementation → fill frontmatter → write 7 body sections → run `cag_validate.py --type agent` → update `agents/README.md` routing table.
  - **Add a new skill** — checklist: create `.github/skills/<name>/` folder → copy `SKILL_TEMPLATE.md` reference implementation → write `description` with both load + skip clauses → put any code in `examples/`/`templates/`/`snippets/` → run `cag_validate.py --type skill`.
  - **Add a new prompt** — checklist: pick verb-noun filename → copy `PROMPT_TEMPLATE.md` reference implementation → fill `loads_skills` and `loads_tools` (every entry must exist) → write binary success criteria → run `cag_validate.py --type prompt`.
- **Linked templates**: all 3 file-type templates.

## Section 7 — End-of-Session CAG Sweep Contract

- **Audience**: AI agents primarily; CAG-Architect specifically.
- **Word budget**: ~200 words.
- **Content**:
  - At every session close, the active driving agent (Manager or solo agent) routes to CAG-Architect with the question: "did anything in this session imply a CAG update?"
  - Triggers for an implied update: new module added (system prompt repo layout?), new tool script added (any agent's `loads_tools`?), new convention discovered and stored to repo memory (any skill's domain knowledge?), agent overlap encountered (routing tables?).
  - CAG-Architect's mandatory final actions: run `python tools/validate/cag_validate.py`; if any change applied, commit with `chore(cag): ...`; append JSONL log entry; update `docs/CHANGELOG.md`.
- **Linked templates**: `AGENT_TEMPLATE.md` (CAG-Architect agent definition).

## Section 8 — Glossary

- **Audience**: both.
- **Word budget**: ~150 words; glossary list format.
- **Terms** (one-sentence definitions):
  - **CAG** — Context Augmented Guidance, the `.github/` layer that customises Copilot for this repo.
  - **Agent** — a role definition file under `.github/agents/` that an executing model embodies.
  - **Skill** — an on-demand domain knowledge module under `.github/skills/<name>/`.
  - **Prompt** — a user-invoked task playbook under `.github/prompts/`.
  - **Persona** — one of six target user profiles the engine and CAG layer serve (EngDev, GameDev, Modder, Player, GameTest, EngTest).
  - **Handover** — a structured packet passed between agents at a routing boundary.
  - **Gate** — a binary check that must pass before a phase or commit is accepted.
  - **Sweep** — the end-of-session CAG-Architect review (see §7).
  - **Cold-start test** — a verification that an agent given only the system prompt can route correctly to the right skill + agent.
  - **Companion file** — a code or template file under a skill folder, referenced from `SKILL.md` instead of inlined.
