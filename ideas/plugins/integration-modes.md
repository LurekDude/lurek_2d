# Plugin Architecture — Integration Modes

## Overview

With `lurek2d-core` extracted as a library crate, Lurek2D can run in four distinct modes:

| Mode | Binary | Owns Window | Who Controls Event Loop | Use Case |
|------|--------|-------------|------------------------|----------|
| **Standalone** | `lurek2d.exe` | Yes | Lurek2D (winit) | Game development, demos |
| **Library** | `libluna2d_core.rlib` | Yes | Lurek2D (winit) | Rust app embeds scripting |
| **Embedded** | `luna2d_core.dll` / `.so` | No | Host app | Python, C#, Electron |
| **Headless** | `lurek2d.exe --headless` | No | Lurek2D (no window) | CI, batch processing, testing |

---

## Mode 1 — Standalone Executable (Current)

This is today's model and the default after the plugin split:

```
┌──────────────────────────────────────────┐
│              lurek2d.exe                   │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │ lurek2d-core (statically linked)    │  │
│  │  engine, math, gfx, audio, ...    │  │
│  └────────────────────────────────────┘  │
│                                          │
│  ┌─────────────────┐  ┌──────────────┐  │
│  │ luna_gamedev.dll │  │ luna_biz.dll │  │
│  │ (plugin)        │  │ (plugin)     │  │
│  └─────────────────┘  └──────────────┘  │
│                                          │
│  main.lua → lurek.init() → event loop     │
└──────────────────────────────────────────┘
```

**Flow**:
1. User runs: `lurek2d.exe demos/my_game`
2. `lurek2d-bin` parses CLI args, loads `conf.toml`
3. `lurek2d-core` initializes winit, wgpu, rodio, creates Lua VM
4. Plugins discovered and loaded
5. `main.lua` executes
6. winit event loop runs until window closes

**Use for**: Game development, demos, interactive applications, digital signage, kiosks.

---

## Mode 2 — Rust Library (Compile-Time Embedding)

A Rust application includes `lurek2d-core` as a dependency in its `Cargo.toml`:

```toml
[dependencies]
lurek2d-core = { path = "../lurek2d/crates/lurek2d-core" }
# or from a registry:
# lurek2d-core = "0.6"
```

```rust
// my_app/src/main.rs
use luna2d_core::{App, Config, create_lua_vm};

fn main() {
    let config = Config::load_from_file("conf.toml").unwrap();
    let app = App::new(&config).unwrap();
    let (state, lua) = create_lua_vm(app.shared_state(), &config.modules).unwrap();

    // Load custom Lua scripts
    lua.load(r#"
        lurek.init(function()
            print("Hello from embedded Lurek2D!")
        end)
    "#).exec().unwrap();

    app.run();  // Blocks on winit event loop
}
```

**Use for**: Rust applications that want Lua scripting + 2D rendering without building
their own engine. Custom game launchers. Game editors (with the engine embedded in a
panel).

**Advantages**:
- Full Rust type safety — direct access to `SharedState`, `RenderCommand`, etc.
- No DLL boundary, no ABI concerns
- Compile-time feature selection via Cargo features

**Disadvantages**:
- Requires Rust toolchain to build
- Tied to specific `lurek2d-core` version at compile time

## Mode 3 — Embedded in Foreign Host (Runtime Linking)

A non-Rust application loads `luna2d_core.dll` (C-ABI shared library) and drives it:

```
┌──────────────────────────────────────────┐
│           Host Application               │
│  (Python, C#, C++, Electron, etc.)       │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │ Host owns the window               │  │
│  │  (e.g. PyQt, WPF, GTK, Electron)  │  │
│  │                                    │  │
│  │  luna2d_core.dll (loaded via FFI)  │  │
│  │   luna2d_init(config)              │  │
│  │   luna2d_eval(lua_code)            │  │
│  │   luna2d_step(dt)                  │  │
│  │   luna2d_render_to_texture(buf)    │  │
│  │   luna2d_shutdown()                │  │
│  └────────────────────────────────────┘  │
│                                          │
│  Host renders the texture to its own     │
│  window/panel using its own graphics API │
└──────────────────────────────────────────┘
```

### C API Surface

```c
// lurek2d.h (generated by cbindgen)
typedef struct LunaContext LunaContext;

// Lifecycle
LunaContext* luna2d_init(const char* config_path);
void         luna2d_shutdown(LunaContext* ctx);

// Lua execution
int          luna2d_load_file(LunaContext* ctx, const char* path);
int          luna2d_eval(LunaContext* ctx, const char* lua_code);

// Frame loop (called by host)
void         luna2d_step(LunaContext* ctx, float dt);
void         luna2d_render(LunaContext* ctx);

// Render-to-texture (for embedded panels)
void         luna2d_render_to_buffer(LunaContext* ctx, uint8_t* rgba_buffer, int width, int height);

// Input forwarding (host sends input events)
void         luna2d_key_event(LunaContext* ctx, int key, int action);
void         luna2d_mouse_event(LunaContext* ctx, float x, float y, int button, int action);

// State queries
float        luna2d_get_fps(LunaContext* ctx);
const char*  luna2d_get_error(LunaContext* ctx);
```

### Python Embedding Example

```python
import ctypes
import time

# Load Lurek2D as a shared library
luna = ctypes.CDLL("luna2d_core.dll")

# Define C function signatures
lurek.luna2d_init.restype = ctypes.c_void_p
lurek.luna2d_init.argtypes = [ctypes.c_char_p]

lurek.luna2d_eval.restype = ctypes.c_int
lurek.luna2d_eval.argtypes = [ctypes.c_void_p, ctypes.c_char_p]

lurek.luna2d_step.argtypes = [ctypes.c_void_p, ctypes.c_float]
lurek.luna2d_render.argtypes = [ctypes.c_void_p]
lurek.luna2d_shutdown.argtypes = [ctypes.c_void_p]

# Initialize
ctx = lurek.luna2d_init(b"conf.toml")

# Load a Lua script
lurek.luna2d_eval(ctx, b"""
    lurek.init(function()
        print("Hello from Python-hosted Lurek2D!")
    end)
""")

# Drive the loop from Python
try:
    last_time = time.time()
    while True:
        now = time.time()
        dt = now - last_time
        last_time = now

        lurek.luna2d_step(ctx, dt)
        lurek.luna2d_render(ctx)

        time.sleep(1/60)
except KeyboardInterrupt:
    lurek.luna2d_shutdown(ctx)
```

### PyO3 Embedding Example (Higher-Level)

```python
# Using PyO3 native module (if we build lurek2d-python crate)
import lurek2d

engine = lurek2d.Engine("conf.toml")
engine.eval("""
    lurek.init(function()
        data = lurek.dataframe.fromCSV("sales.csv")
    end)
""")

# Get data back from Lua → Python
result = engine.eval_return("return data:to_json()")
import json
sales_data = json.loads(result)

engine.shutdown()
```

### C# / .NET Embedding Example

```csharp
using System.Runtime.InteropServices;

public class Lurek2D {
    [DllImport("luna2d_core.dll")]
    static extern IntPtr luna2d_init(string config);

    [DllImport("luna2d_core.dll")]
    static extern int luna2d_eval(IntPtr ctx, string code);

    [DllImport("luna2d_core.dll")]
    static extern void luna2d_step(IntPtr ctx, float dt);

    [DllImport("luna2d_core.dll")]
    static extern void luna2d_render(IntPtr ctx);

    [DllImport("luna2d_core.dll")]
    static extern void luna2d_shutdown(IntPtr ctx);

    private IntPtr ctx;

    public void Init(string config = "conf.toml") {
        ctx = luna2d_init(config);
    }

    public void Eval(string luaCode) {
        luna2d_eval(ctx, luaCode);
    }

    public void Update(float dt) {
        luna2d_step(ctx, dt);
        luna2d_render(ctx);
    }

    public void Shutdown() {
        luna2d_shutdown(ctx);
        ctx = IntPtr.Zero;
    }
}

// Usage in a WPF application:
var luna = new Lurek2D();
lurek.Init("conf.toml");
lurek.Eval(@"
    lurek.init(function()
        lurek.render.setBackgroundColor(0.2, 0.2, 0.3)
    end)
");

// In your WPF render loop:
CompositionTarget.Rendering += (s, e) => {
    lurek.Update(0.016f);
};
```

**Use for**: Embedding Lurek2D rendering in a larger application. Python data science
tools that need visualization. C# business apps that need interactive dashboards.
Electron apps that want native 2D rendering performance.

**Challenges**:
- Window ownership: who creates the GPU surface? Two options:
  1. Lurek2D creates its own window (simple but can't embed in host panels)
  2. Host passes a window handle / surface — Lurek2D renders into it (complex but flexible)
- Thread safety: `luna2d_step` and `luna2d_render` must be called from the same thread
  (wgpu requirement). The host must ensure this.
- Error handling: `luna2d_get_error()` returns the last error string. Host checks after
  each call.

## Mode 4 — Headless (No Window, No GPU)

```
┌──────────────────────────────────────────┐
│         lurek2d.exe --headless            │
│                                          │
│  No winit window                         │
│  No wgpu device                          │
│  No audio device                         │
│                                          │
│  Lua VM with:                            │
│   ✓ lurek.math, lurek.data, lurek.filesystem        │
│   ✓ lurek.dataframe, lurek.pipeline        │
│   ✓ lurek.graph, lurek.ai (logic only)     │
│   ✓ lurek.serial, lurek.event              │
│   ✗ lurek.render, lurek.audio, lurek.window    │
│   ✗ lurek.input.keyboard, lurek.input.mouse            │
└──────────────────────────────────────────┘
```

**Flow**:
1. `lurek2d.exe --headless script.lua`
2. No window or GPU initialization
3. Lua VM created with only non-visual modules
4. `script.lua` runs to completion, then exits

**Use for**: CI/CD test pipelines, batch data processing, server-side game logic
(authoritative multiplayer), code linting/validation, report generation.

```lua
-- headless_report.lua — runs without a window
local df = lurek.dataframe.fromCSV("sales_2024.csv")
local summary = df:groupBy("region"):sum("revenue")
lurek.filesystem.write("report.json", summary:to_json())
print("Report generated: " .. summary:rowCount() .. " regions")
```

## Comparison Matrix

| Feature | Standalone | Library | Embedded | Headless |
|---------|-----------|---------|----------|----------|
| Window | Lurek2D | Lurek2D | Host | None |
| GPU rendering | Yes | Yes | Yes (host surface) | No |
| Audio | Yes | Yes | Yes | No |
| Input | Yes | Yes | Host forwards | No |
| Lua scripting | Yes | Yes | Yes | Yes |
| Plugin DLLs | Yes | N/A (compile-time) | Possible | Yes |
| File I/O | Yes (sandboxed) | Yes | Yes | Yes |
| Who builds | End user | Rust developer | FFI consumer | CI/script runner |
| Complexity | Low | Medium | High | Low |

---

## Implementation Priority

1. **Standalone** — already exists; plugin split enhances it (Phase 2)
2. **Headless** — partially exists (`create_test_lua_vm` in tests); formalize with
   `--headless` flag (Phase 2, low effort)
3. **Library** — natural result of workspace split; `lurek2d-core` becomes a usable
   crate (Phase 1 side effect)
4. **Embedded** — requires C API surface, `cbindgen`, and careful thread/window
   ownership design (Phase 5)
