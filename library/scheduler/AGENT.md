## Summary

Pure-Lua **coroutine scheduler** â€” a cooperative task runner that integrates
with the engine's update loop. Tasks are coroutine bodies that receive a
`yield(seconds)` helper; calling `yield(n)` suspends the task for `n` seconds
of game time, where game time advances by whatever `dt` the host passes to
`Scheduler:update(dt)`.

This is **distinct** from the engine `lurek.timer.Scheduler` userdata. The
engine scheduler is a wall-clock one-shot/interval registry (`:after(delay,
fn)`, `:every(interval, fn)`, `:cancel(id)`). `library.scheduler` is a
**coroutine** scheduler driven by your `dt` accumulator â€” useful for cutscenes,
multi-step gameplay sequences, and tests that need deterministic stepping.

## Rename History

This module was previously named `library.patterns`. It was renamed to
`library.scheduler` in 0.6.0 because the old name collided with the unrelated
`lurek.patterns` Rust namespace (`EventBus`, `ObjectPool`, `CommandStack`,
`SimpleState`, â€¦). The original folder `library/patterns/` now contains
a deprecation stub that forwards to this module via `require("library.scheduler")`.

## Notes

- `lurek.log` is consulted **at module load** (best-effort `pcall`) and
  silently degrades to no-op stubs in headless test VMs.
- Iteration guard: `update()` will break out of its inner loop once
  `max_iterations` resumes have happened in a single tick. This guards
  against tasks that `yield(0)` in a tight loop.
- All task ids are monotonic â€” they are never recycled inside a Scheduler
  instance.
- See also `lurek.timer.Scheduler` for wall-clock `:after`/`:every`
  scheduling (different semantics).
